# 6. Limitations

We identify ten limitations of this study, grouped by their impact on the interpretation of results.

**Correlational design.** All findings are associational. No causal intervention was performed. The association between a behavioral feature and task success may be confounded by task difficulty: easier tasks may simultaneously elicit better reasoning structure and produce more successful outcomes. The retraction of reasoning_to_action_alignment -- which appeared significant in pooled analysis but was inflated by between-repo confounding -- illustrates how correlational findings can mislead. Phase 1 must test whether gating on these features changes outcomes before any causal claims can be made.

**No predictive evaluation.** We report rank correlations and permutation tests, not predictions. There is no held-out test set, no area under the ROC curve, no calibration analysis, and no out-of-sample evaluation. The interaction effect (73% pass in the high-deliberation, high-precision quadrant) is an observational association, not a validated prediction rule. Claims about "prediction viability" or "reliability prediction" are premature until Phase 1 produces a prospective evaluation.

**Model-specificity.** Linguistic features do not transfer across models. The confirmed polarity flip on contrastive marker density and the entirely disjoint sets of CWC-surviving features for Sonnet and Codex mean that our findings are specific to these two model versions. We cannot claim that these associations would hold for other models, or even for future versions of the same models. Only the two structural features (fail_streak_max, early_error_rate) showed consistent direction across both architectures.

**Narrow domain.** All tasks are drawn from SWE-bench Lite, a corpus of real-world Python bug-fix tasks from open-source repositories. The behavioral features we extract are specific to code repair workflows (file editing, test running, error handling). Whether similar features would be informative for other agent tasks -- web browsing, data analysis, multi-step planning -- is unknown.

**Sample size per model-repo cell.** The within-repo analyses that form the backbone of our statistical approach operate on Sonnet/sympy (n=87), Sonnet/django (n=44), Codex/sympy (n=84), and Codex/django (n=71). The interaction analysis uses quadrant splits with approximately 22 runs per cell. These sample sizes provide adequate power for medium-to-large effects but may miss smaller associations. A formal power analysis was not conducted prior to data collection.

**Thinking block format differences.** Sonnet's extended thinking blocks and Codex's reasoning items are architecturally different. Per-call reasoning text coverage differs substantially (28.8% for Sonnet vs. 60.1% for Codex). Cross-model comparisons of features extracted from these different formats must be interpreted with caution. The confirmed polarity flip could partly reflect extraction artifacts rather than genuine differences in how the models use language.

**Codex convergence.** Codex's sympy pass rate improved by 10.3 percentage points from early to late runs. While within one standard deviation of random variation for the sample size, this trend could reflect infrastructure stabilization or task-ordering effects. If the improvement is genuine, early Codex runs are biased downward, which would affect the Codex-specific feature associations reported for sympy.

**Within-repo permutation scope.** The permutation test shuffles success labels within each repository, controlling for between-repo confounds (different pass rates, different codebases). It does not control for within-repo task difficulty variation. A within-task permutation test -- shuffling labels among replicates of the same task -- is the stronger design but requires sufficient mixed-outcome tasks. Our within-task analysis is limited to 7 Sonnet tasks (35 runs) and 5 Codex tasks (18 runs) with mixed outcomes. No task has mixed outcomes in both models simultaneously, precluding cross-model within-task permutation.

**Unmatched task sets.** Five tasks were run on Sonnet but not Codex. All cross-model comparisons use the 70 shared tasks, but the five Sonnet-only tasks could bias Sonnet-specific feature estimates if those tasks have unusual behavioral profiles.

**Single labeler.** All correctness labels were assigned by a single researcher using a blinded dual-pass protocol (independent validation via test replay, not agent self-report). While the validation is automated and deterministic once the test suite is specified, the decision to classify ambiguous cases (e.g., partial test passage, infrastructure failures) reflects one person's judgment. Four runs were excluded as unrecoverable. Inter-rater reliability was not assessed.
