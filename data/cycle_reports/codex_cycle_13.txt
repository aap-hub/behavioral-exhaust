1. P0: `task_success` is not stable ground truth. [`src/runner.py:326`](file:///Users/al/projects/uga-harness/src/runner.py#L326) writes it as the original in-workspace validation outcome, but [`src/docker_validate.py:127`](file:///Users/al/projects/uga-harness/src/docker_validate.py#L127) and [`src/swebench_validate.py:263`](file:///Users/al/projects/uga-harness/src/swebench_validate.py#L263) overwrite the same column with later Docker/SWE-bench results. 2. P1: The “atomic” run insert path is not atomic. [`src/runner.py:374`](file:///Users/al/projects/uga-harness/src/runner.py#L374) and [`src/analysis.py:408`](file:///Users/al/projects/uga-harness/src/analysis.py#L408) wrap writes in `BEGIN`, but [`src/db.py:218`](file:///Users/al/projects/uga-harness/src/db.py#L218), [`src/db.py:247`](file:///Users/al/projects/uga-harness/src/db.py#L247), and [`src/db.py:288`](file:///Users/al/projects/uga-harness/src/db.py#L288) call `commit()` inside helpers. A later failure can leave partially written runs/tool_calls despite the caller believing rollback will restore consistency.

3. P1: Patch extraction silently drops failed `Edit` replays. [`src/swebench_validate.py:102`](file:///Users/al/projects/uga-harness/src/swebench_validate.py#L102) replays edits with `if old in modified: replace(...); else: no-op`. If the captured `old_string` mismatches, the change disappears from the generated patch with no error, biasing validation toward false negatives / “no patch extracted”.

4. P1: Bash read-only classification has false negatives for mutating commands. In [`src/trace_collector.py:174`](file:///Users/al/projects/uga-harness/src/trace_collector.py#L174), any pipeline containing `&&` or `;` skips pipeline analysis, then [`src/trace_collector.py:196`](file:///Users/al/projects/uga-harness/src/trace_collector.py#L196) classifies by first token only. `cat a | tee b && rm c` is read-only because `cat` is whitelisted. Also `xargs` is globally whitelisted at [`src/trace_collector.py:61`](file:///Users/al/projects/uga-harness/src/trace_collector.py#L61), so plain `xargs rm < files` is misclassified read-only.

5. P2: Analysis summary overstates analyzed sample size. [`src/analysis.py:168`](file:///Users/al/projects/uga-harness/src/analysis.py#L168) counts all SWE-bench runs, but pass/fail sums exclude `NULL task_success`; the DB currently has 3 such runs. The printed denominator and returned `n_runs` include unlabeled runs, so the summary is internally inconsistent and can misstate analysis coverage.
