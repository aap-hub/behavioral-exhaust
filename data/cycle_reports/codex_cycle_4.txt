P0. [`src/db.py`](/Users/al/projects/uga-harness/src/db.py#L45) defines `runs.task_success` as “`validation_command exit 0`”, but both [`src/docker_validate.py`](/Users/al/projects/uga-harness/src/docker_validate.py#L127) and [`src/swebench_validate.py`](/Users/al/projects/uga-harness/src/swebench_validate.py#L265) overwrite that same column with external Docker/SWE-bench outcomes. P1. The “single transaction” claim is false. [`src/runner.py`](/Users/al/projects/uga-harness/src/runner.py#L374) does `BEGIN`, but [`insert_run`](/Users/al/projects/uga-harness/src/db.py#L197) and [`insert_tool_call`](/Users/al/projects/uga-harness/src/db.py#L225) each call `commit()`. Same bug in [`analysis.reparse_all_runs`](/Users/al/projects/uga-harness/src/analysis.py#L401). Any later failure can leave a partially written run / partially replaced `tool_calls` set.

P1. Bash state-modifying classification is unsound for compound commands. In [`src/trace_collector.py`](/Users/al/projects/uga-harness/src/trace_collector.py#L174), pipelines are handled, but commands containing `&&` or `;` fall through to first-token logic only. Example: `git diff; rm foo` is classified read-only because `git diff` wins. Bare `xargs` is also whitelisted as read-only at [`src/trace_collector.py`](/Users/al/projects/uga-harness/src/trace_collector.py#L61), so `xargs rm` is dropped. This causes silent undercounting and missing state-modifying records.

P1. Patch extraction silently loses real changes. [`src/swebench_validate.py`](/Users/al/projects/uga-harness/src/swebench_validate.py#L35) only replays `Write`/`Edit`, ignoring file mutations done through Bash (`sed -i`, redirects, `mv`, `git apply`, scripts, etc.). It also silently skips failed edit replays at [`src/swebench_validate.py`](/Users/al/projects/uga-harness/src/swebench_validate.py#L106). Docker/SWE-bench validation can therefore score an incomplete patch and then overwrite `task_success`.

P2. `prior_failure_streak` is biased downward. In [`src/analysis.py`](/Users/al/projects/uga-harness/src/analysis.py#L99), missing or unparseable `tool_result_json` is treated as success (`True`). Unknown should not become success; this silently suppresses prior failures in extracted features.

P2. `call_level_correlations` still computes p-values on non-independent rows. [`src/analysis.py`](/Users/al/projects/uga-harness/src/analysis.py#L294) acknowledges run-level labels reused across many calls, but still reports Spearman `p` on call rows. The inference is invalid/inflated, not just noisy.
