**Findings**
1. `P0` Codex runs do not have a comparable validation path to Sonnet. The runner writes `task_success` from live-workspace validation, not fresh-copy replay, and it never writes `validation_source`. Its post-hoc path reuses the Sonnet-only edit replayer, which only extracts `Write`/`Edit` tool uses and returns `no edits found` on Codex-style `command_execution` traces. Alternative validators in the repo are also hard-coded to `model_version='sonnet'`. This means the locked protocol’s stated outcome variable cannot be produced for Codex as written. [codex_runner.py:616](src/codex_runner.py#L616) [codex_runner.py:797](src/codex_runner.py#L797) [independent_validate.py:59](src/independent_validate.py#L59) [independent_validate.py:273](src/independent_validate.py#L273) [protocol_tasks.json:3](data/protocol_tasks.json#L3) [swebench_validate.py:175](src/swebench_validate.py#L175) [docker_validate.py:176](src/docker_validate.py#L176)

2. `P0` The existing analysis path is not Codex-safe. `analysis.py` hard-codes Sonnet filters, so it excludes Codex entirely. `tier2_features.py` has the opposite problem: it does not filter by model at all and deduplicates by `task_id`, so once Codex runs the same 71 tasks it will silently replace or mix Sonnet runs. `analysis.py --reparse` is worse: it reparses all runs with the Claude trace parser and would regenerate zero tool calls for Codex rows. [analysis.py:169](src/analysis.py#L169) [analysis.py:209](src/analysis.py#L209) [analysis.py:311](src/analysis.py#L311) [analysis.py:382](src/analysis.py#L382) [tier2_features.py:420](src/tier2_features.py#L420)

3. `P0` `parse_codex_stream()` does not attach reasoning the way the Sonnet parser does. It clears pending reasoning and agent-message buffers after every `command_execution`, including read-only commands, so reasoning before `cat`/`rg`/`git diff` is lost before the later state-modifying command. The Sonnet parser explicitly preserves reasoning across non-state-modifying calls. I verified this with a minimal Codex-style stream: reasoning before `cat foo.py` is dropped before a later write command. [codex_runner.py:392](src/codex_runner.py#L392) [trace_collector.py:475](src/trace_collector.py#L475) [trace_collector.py:551](src/trace_collector.py#L551)

4. `P0` Reasoning and polished messages are captured, but not as comparable or truly separate channels. Codex merges them into one `reasoning_text` blob with `---THINKING---` / `---MESSAGE---` markers, and all current feature extraction reads that merged field directly. Sonnet stores only pre-tool reasoning text. So Codex linguistic features would be computed on a different object than Sonnet’s. [codex_runner.py:324](src/codex_runner.py#L324) [analysis.py:92](src/analysis.py#L92) [analysis.py:149](src/analysis.py#L149) [tier2_features.py:364](src/tier2_features.py#L364)

5. `P0` The runner CLI does not accept the locked protocol file shape. `codex_runner.py` expects a raw JSON array for `batch_file`, but `data/protocol_tasks.json` is a metadata object with a `tasks` field. If you pass the locked protocol file directly, it will iterate keys like `created` and fail on the first `load_task()` lookup. I verified the file loads as a 5-key dict. [codex_runner.py:861](src/codex_runner.py#L861) [codex_runner.py:504](src/codex_runner.py#L504) [protocol_tasks.json:1](data/protocol_tasks.json#L1) [protocol.py:54](protocol.py#L54)

6. `P1` The claimed “single transaction for atomicity” is false. `run_codex_task()` does `BEGIN`, but `db.insert_run()` and `db.insert_tool_call()` call `commit()` internally, so partial run rows and partial tool-call prefixes can persist if a later insert fails. The per-row commits also increase SQLite writer churn under 3-5 concurrent processes. Workspace conflicts are unlikely because `mkdtemp()` gives unique workspaces; DB contention is the real concurrency risk. [codex_runner.py:667](src/codex_runner.py#L667) [db.py:225](src/db.py#L225) [db.py:257](src/db.py#L257)

7. `P1` Timed-out Codex runs are handled differently from timed-out Sonnet runs. Sonnet skips validation after timeout; Codex logs “Skipping validation” and then still validates the timed-out workspace. That can create `task_success` labels for runs that the Sonnet path would leave unlabeled. [codex_runner.py:621](src/codex_runner.py#L621) [runner.py:328](src/runner.py#L328)

8. `P1` The Codex JSON parser can duplicate reasoning/message text if `item.started` carries text. It ingests both `item.started` and `item.completed` for `reasoning` and `agent_message`, but only guards start events for `command_execution`. I verified a started+completed reasoning pair produces duplicated `reasoning_text`. [codex_runner.py:276](src/codex_runner.py#L276) [codex_runner.py:297](src/codex_runner.py#L297)

**Scope Answers**
1. Comparable to Sonnet: schema mostly yes; validation method and reasoning-text semantics no.
2. `parse_codex_stream()` mapping: command classification is mostly right after unwrapping, but reasoning attribution is not comparable and started/completed handling can duplicate text.
3. Validation before cleanup: yes. [codex_runner.py:616](src/codex_runner.py#L616) [codex_runner.py:738](src/codex_runner.py#L738)
4. `classify_state_modifying()` on unwrapped Codex commands: mostly yes. [codex_runner.py:313](src/codex_runner.py#L313) [trace_collector.py:133](src/trace_collector.py#L133)
5. Reasoning and message streams: captured, but merged into one field and not first-class separate columns.
6. Existing analysis pipeline by `model_version`: no.
7. Data integrity risks: yes, mainly partial DB writes and destructive reparse on mixed-model DBs.
8. Parallel 3-5 runners: workspace isolation is fine; SQLite contention risk is real.
9. Codex JSON parsing bugs: yes, at least started/completed duplication and silent malformed-line drops in `parse_codex_stream()`.
10. Validation fairness: no. Live-workspace validation is not comparable to Sonnet’s replay-based validated labels.

Do not launch as-is. The hard blockers are: no comparable validated outcome for Codex, non-comparable reasoning capture, analysis code that either excludes Codex or mixes it with Sonnet, and the protocol-file input mismatch if that file is the launch artifact.
