# Codex Audit — Cycle 9
# 2026-03-29T06:28:16.318857+00:00

1. P1: The “single transaction” guarantees are broken. `runner.py` and `analysis.py` call `BEGIN`/`ROLLBACK`, but the DB helpers commit inside each insert/update, so a mid-run failure can leave a persisted `runs` row plus a partial subset of `tool_calls` or features already committed. Silent partial writes become unrecoverable state, not atomic failures. 2. P1: `prior_failure_streak` silently undercounts failures. Missing or unparseable `tool_result_json` is treated as success, so dropped tool results and parse errors reset the streak instead of preserving uncertainty/failure. That corrupts a core feature without surfacing an error. [src/analysis.py](/Users/al/projects/uga-harness/src/analysis.py#L99)

3. P1: Validation can silently score the wrong patch. `extract_patch_from_stream()` only replays `Write`/`Edit`, ignores state-modifying `Bash`, and for `Edit` silently skips an edit if `old_string` is not found. That can produce incomplete patches, dropped file changes, and wrong `task_success` labels in both Docker and SWE-bench validation. [src/swebench_validate.py](/Users/al/projects/uga-harness/src/swebench_validate.py#L23) [src/swebench_validate.py](/Users/al/projects/uga-harness/src/swebench_validate.py#L102) [src/docker_validate.py](/Users/al/projects/uga-harness/src/docker_validate.py#L98)

4. P1: Bash state-modifying classification is wrong for compound commands. The classifier only handles pipes specially; `&&`/`;` chains fall through to first-token logic, so commands like `cat foo && rm foo` or `git diff && git apply patch` can be marked read-only and never recorded/gated. Standalone `xargs` is also whitelisted as read-only even though it commonly executes mutating commands. [src/trace_collector.py](/Users/al/projects/uga-harness/src/trace_collector.py#L172) [src/trace_collector.py](/Users/al/projects/uga-harness/src/trace_collector.py#L196) [src/trace_collector.py](/Users/al/projects/uga-harness/src/trace_collector.py#L217)

5. P2: Read-only `git fetch` is classified as state-modifying. `_GIT_MUTATING_SUBCOMMANDS` includes `fetch`, which mutates local refs but not the task workspace. If your taxonomy is “bash read-only vs state-modifying” in the sense of repository/filesystem risk, this overcounts state-modifying calls and contaminates the gating label. [src/trace_collector.py](/Users/al/projects/uga-harness/src/trace_collector.py#L82)

6. P2: The current DB looks internally consistent on coarse counts only. `SUM(total_state_modifying_calls)=COUNT(tool_calls)=1345`, and no run has `total_tool_calls < total_state_modifying_calls`; that does not catch the partial-write bug above because helper-level commits can still leave logically incomplete runs that satisfy those invariants.
