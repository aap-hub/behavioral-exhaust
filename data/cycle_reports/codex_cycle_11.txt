P1. Fake atomic writes: `runner.py` and `analysis.py` wrap inserts in `BEGIN`, but the helpers they call `commit()` on every row, so a later failure can leave a run partially written despite the surrounding rollback path. This risks orphaned/incomplete `runs`/`tool_calls` state. P1. Validation patch reconstruction silently corrupts edits: `extract_patch_from_stream()` replays `Edit` by `modified.replace(old, new, 1)` from the top of the file. If the same `old_string` appears multiple times, it can patch the wrong occurrence; if an earlier edit changes the context, later edits can be skipped with no error. Both Docker and SWE-bench validators then score the wrong patch. [src/swebench_validate.py](/Users/al/projects/uga-harness/src/swebench_validate.py#L102) [src/docker_validate.py](/Users/al/projects/uga-harness/src/docker_validate.py#L98)

P1. Validation drops state-modifying Bash entirely. Patch extraction only replays `Write`/`Edit`, so file mutations done via Bash (`mv`, `cp`, heredocs, `sed -i`, `git apply`, etc.) vanish from the reconstructed patch. That yields false “no patch extracted” or wrong unresolved labels. [src/swebench_validate.py](/Users/al/projects/uga-harness/src/swebench_validate.py#L37) [src/docker_validate.py](/Users/al/projects/uga-harness/src/docker_validate.py#L98)

P1. `task_success` is semantically overwritten after the run. Schema says it is the in-workspace `validation_command` result, but both validation scripts rewrite the same column with Docker/SWE-bench outcomes, destroying provenance and mixing incompatible success definitions in later analysis. That makes feature totals and “total_calls” non-comparable within the same correlation table. P2. Failure-streak extraction biases toward success on missing/unparseable tool results.
