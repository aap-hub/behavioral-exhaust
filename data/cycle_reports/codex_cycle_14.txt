# Codex Audit — Cycle 14
# 2026-03-29T09:18:40.953038+00:00

1. P0: The “atomic” DB writes are not atomic. [`src/db.py`](src/db.py#L218) and [`src/db.py`](src/db.py#L247) call `conn.commit()` inside `insert_run`/`insert_tool_call`, which breaks the explicit transaction in [`src/runner.py`](src/runner.py#L374) and the delete+reinsert transaction in [`src/analysis.py`](src/analysis.py#L410). A mid-batch failure can leave partial runs or partially reparsed `tool_calls` committed.

2. P0: Patch reconstruction silently drops real edits after a `Write`. In [`src/swebench_validate.py`](src/swebench_validate.py#L95), if a file has both `Write` and later `Edit` calls, the `full_write` branch wins and `changes['edits']` is ignored. Validation then runs against the wrong patch and can overwrite `runs.task_success` with a false result in [`src/swebench_validate.py`](src/swebench_validate.py#L265) and [`src/docker_validate.py`](src/docker_validate.py#L129).

3. P1: Patch reconstruction also silently drops failed `Edit` applications. [`src/swebench_validate.py`](src/swebench_validate.py#L106) does `replace(old, new, 1)` only if `old in modified`; otherwise it skips the edit with no error. That corrupts validation patches without any signal.

4. P1: `prior_failure_streak` is biased downward by treating missing/unparseable tool results as success. [`src/analysis.py`](src/analysis.py#L103) and [`src/analysis.py`](src/analysis.py#L108) append `True` on missing/invalid `tool_result_json`. That silently erases preceding failures from the feature.

5. P1: Standalone `xargs` is misclassified as read-only. [`src/trace_collector.py`](src/trace_collector.py#L61) includes `xargs` in `READ_ONLY_COMMANDS`, so `xargs rm`, `xargs mv`, etc. are counted as non-state-modifying unless they happen to be in a pipe caught by the separate pipe logic. This undercounts `total_state_modifying_calls` and drops decision records.

6. P1: Unknown `git` subcommands default to read-only. [`src/trace_collector.py`](src/trace_collector.py#L217) returns read-only for any `git <subcommand>` not in `_GIT_MUTATING_SUBCOMMANDS`. Mutating commands omitted from that list are silently misclassified and excluded from analysis.

7. P2: Task-level analysis excludes runs with zero extracted `tool_calls`. [`src/analysis.py`](src/analysis.py#L199) inner-joins `tool_calls` to `runs`, so runs with no state-modifying calls disappear entirely from correlation analysis. That biases task-level aggregates toward trajectories that already made state changes.
