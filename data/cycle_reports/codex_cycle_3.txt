# Codex Audit — Cycle 3
# 2026-03-29T04:00:02.262515+00:00

P1. Claimed atomic insert is not atomic. [`src/runner.py:374`](src/runner.py#L374) starts `BEGIN`, but [`src/db.py:197`](src/db.py#L197) and [`src/db.py:225`](src/db.py#L225) each call `conn.commit()`. If any later `tool_calls` insert fails, the `runs` row and earlier calls are already persisted, so rollback cannot restore consistency.

P1. Validation rewrites `runs.task_success` with a different meaning. [`src/docker_validate.py:127`](src/docker_validate.py#L127) and [`src/swebench_validate.py:263`](src/swebench_validate.py#L263) overwrite the original runner outcome (`validation_command` exit code) instead of storing a separate validated field. That silently corrupts longitudinal analysis because `task_success` stops meaning one thing across rows.

P1. Patch extraction can silently produce the wrong patch. [`src/swebench_validate.py:42`](src/swebench_validate.py#L42) only replays `Write`/`Edit`, so any state-changing `Bash` edits are omitted. Then [`src/swebench_validate.py:106`](src/swebench_validate.py#L106) ignores `Edit` operations whose `old_string` is not found instead of failing. Result: validation can mark the wrong patch unresolved/resolved without any indication the replay drifted.

P1. Bash state-modifying classification has clear false negatives. [`src/trace_collector.py:61`](src/trace_collector.py#L61) treats standalone `xargs` as read-only, so `xargs rm`, `xargs sed -i`, etc. are misclassified. [`src/trace_collector.py:174`](src/trace_collector.py#L174) only inspects pipelines when there is no `&&` or `;`, so `cat f | tee out && echo ok` falls through and gets labeled read-only from the first token.

P2. `python -c` / `node -e` classification is too permissive. [`src/trace_collector.py:111`](src/trace_collector.py#L111) says these are read-only only when they print, but [`src/trace_collector.py:124`](src/trace_collector.py#L124) actually returns read-only for any snippet that avoids a short denylist. Arbitrary side effects outside that list are counted as read-only.

P2. `prior_failure_streak` is biased downward by treating missing/unparseable prior results as success. [`src/analysis.py:103`](src/analysis.py#L103) and [`src/analysis.py:110`](src/analysis.py#L110) append `True` on parse failure or missing result, which silently erases prior failures from the derived feature.
