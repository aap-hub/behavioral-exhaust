# Codex Audit — Cycle 12
# 2026-03-29T08:19:20.405262+00:00

1. `P1` Atomicity is broken: [`src/db.py` lines 197-251](src/db.py#L197) call `conn.commit()` inside `insert_run()` and `insert_tool_call()`, so the outer `BEGIN/ROLLBACK` in [`src/runner.py` lines 374-388](src/runner.py#L374) and [`src/analysis.py` lines 410-425](src/analysis.py#L410) cannot protect a run-level batch. Any later insert failure can leave a persisted `runs` row with missing `tool_calls`, or a reparsed run with old calls deleted and only some new calls reinserted.

2. `P1` Infrastructure-failure runs silently undercount work: [`src/runner.py` lines 398-426](src/runner.py#L398) saves `raw_partial` but hardcodes `total_tool_calls = 0`, `total_state_modifying_calls = 0`, and inserts no parsed `tool_calls`. If Claude emitted usable partial trace before the exception, those records are dropped and miscounted.

3. `P1` Validation labels can be corrupted by patch reconstruction: [`src/swebench_validate.py` lines 23-50](src/swebench_validate.py#L23) replays every `Write`/`Edit` tool use without checking whether the tool actually succeeded, and [`src/swebench_validate.py` lines 102-109](src/swebench_validate.py#L102) silently skips an edit when `old_string` is not found. That can fabricate a patch that does not match the agent’s final workspace. [`src/docker_validate.py` line 23](src/docker_validate.py#L23) inherits the same bug.

4. `P2` Task-level analysis silently drops zero-state-modification runs: [`src/analysis.py` lines 199-209](src/analysis.py#L199) builds task stats via `tool_calls JOIN runs`, so any run with no extracted `tool_calls` disappears entirely from correlations. That biases task-level success/feature relationships toward runs that mutated state at least once.

5. `P2` Call-level statistics are still invalid despite the warning: [`src/analysis.py` lines 299-355](src/analysis.py#L299) computes Spearman and `p` on per-call rows while reusing the same run-level `task_success` label across all calls in a run. That is pseudoreplication; the reported `p` values and effective sample size are wrong.

6. `P2` Bash classification has a false-negative hole: [`src/trace_collector.py` lines 52-67](src/trace_collector.py#L52) whitelist `xargs` as read-only, and [`src/trace_collector.py` lines 196-197](src/trace_collector.py#L196) immediately returns `False` for standalone `xargs ...`. `xargs rm`, `xargs mv`, etc. will be mislabeled read-only unless they happen to appear inside the special pipe branch.
