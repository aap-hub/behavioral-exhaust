**Findings**
1. `P0` The analysis code does not test the 10 claimed features. `protocol.py` defines 10 features, but the task-level analysis only uses 6 reasoning features, drops `step_index_normalized`/`prior_failure_streak`/`retry_count`/`tool_switch_rate`, and substitutes `total_calls`, `*_sum`, `*_max`, and `calls_with_reasoning`. The call-level analysis also only uses 6 features. The reported “backtrack_count rho=...” claim is not what this code path computes. [protocol.py:184](protocol.py#L184) [analysis.py:203](src/analysis.py#L203) [analysis.py:257](src/analysis.py#L257) [analysis.py:303](src/analysis.py#L303)

2. `P0` The outcome variable is not restricted to independent validation. `runner.py` writes `task_success` from ad hoc in-workspace validation on the agent-edited workspace, and both correlation queries accept any non-null `task_success` with no `validation_source` filter. That means the dependent variable can be the agent’s own workspace test result, not fresh-copy replay or SWE-bench evaluation. [runner.py:326](src/runner.py#L326) [runner.py:351](src/runner.py#L351) [analysis.py:203](src/analysis.py#L203) [analysis.py:303](src/analysis.py#L303) [protocol_tasks.json:3](data/protocol_tasks.json#L3)

3. `P0` `trace_validate.py` can overwrite stronger labels with agent self-report. It accepts medium/high-confidence labels from the agent’s own test output and writes them back into `runs.task_success`. Its “preserve Docker ground truth” guard only checks for `swebench-docker-validated` in `notes`, but the official validators currently write `validation_source='docker-validated: ...'` or `notes='swebench-validated: ...'`, so the guard never matches. This is direct label leakage. 4. `P0` `protocol.py` no longer implements the locked 71-task experiment. It still hard-codes a 10-task, 10-replicate protocol, and `run_protocol()` defaults to `REPLICATES_PER_TASK=10` because `data/protocol_tasks.json` has 71 tasks but no `replicates_per_task`. Running the entry point would try to fill each locked task to 10 runs, not execute the stated one-pass 71-task protocol. 5. `P0` The DB helper commit logic can silently lose writes. `insert_run()`, `insert_tool_call()`, and `update_features()` only call `commit()` when `not conn.in_transaction`, but SQLite flips `in_transaction` to `True` immediately after DML. I verified that with in-memory sqlite here. As written, standalone failure-path `insert_run()` calls and feature extraction updates can roll back on close. 7. `P1` Validation provenance is inconsistent and lossy. `independent_validate.py` only writes `validation_source` when the new label differs from the old `task_success`, so confirmed matches leave no provenance and will be revalidated forever. 8. `P1` The significance reporting is inflated. `analysis.py` prints raw Spearman p-values with no multiple-testing correction, and the call-level table still prints p-values/stars on raw tool-call counts even though the code itself says the effective `n` is runs, not calls. A claimed `p=0.017` is not family-wise significant under Bonferroni for 10 features (`0.005`). 11. `P1` There is a real path-traversal / arbitrary file access issue in replay and patch generation. In `swebench_validate.py`, the same trust can read arbitrary host files and include them in generated diffs. Those paths come from raw model trace data, not a trusted manifest. 13. `P2` The critical path is barely tested. **Limitations**
I attempted the requested live DB queries against [data/uga.db](data/uga.db), including `sqlite3` and Python’s `sqlite3` in explicit read-only mode, but this sandbox denies opening the database file even though the file is readable as bytes. So I could not confirm the current `.schema` or live row counts from the database itself.
