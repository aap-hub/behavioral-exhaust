# Codex Audit Fix Checklist

Deduplicated from 18 Codex audit cycles (40 total findings -> 40 unique)


## P0 Findings

- [ ] **P0-01** (cycle 1): The “atomic” write paths are not atomic. [`src/db.py`](src/db.py#L197) and [`src/db.py`](src/db.py#L225) call `conn.commit()` internally, so the outer transactions in [`src/runner.py`](src/runner.py#L374) and [`src/analysis.py`](src/analysis.py#L411) cannot roll back a partial run/reparse. A failure after `insert_run()` or after some `insert_tool_call()`s leaves silent partial state: orphaned `runs` rows, partially replaced `tool_calls`, and misaligned aggregates.
- [ ] **P0-02** (cycle 13): `task_success` is not stable ground truth. [`src/runner.py:326`](file://src/runner.py#L326) writes it as the original in-workspace validation outcome, but [`src/docker_validate.py:127`](file://src/docker_validate.py#L127) and [`src/swebench_validate.py:263`](file://src/swebench_validate.py#L263) overwrite the same column with later Docker/SWE-bench results. That silently corrupts provenance and makes downstream analysis mix different labels.
- [ ] **P0-03** (cycle 14): The “atomic” DB writes are not atomic. [`src/db.py`](src/db.py#L218) and [`src/db.py`](src/db.py#L247) call `conn.commit()` inside `insert_run`/`insert_tool_call`, which breaks the explicit transaction in [`src/runner.py`](src/runner.py#L374) and the delete+reinsert transaction in [`src/analysis.py`](src/analysis.py#L410). A mid-batch failure can leave partial runs or partially reparsed `tool_calls` committed.
- [ ] **P0-04** (cycle 14): Patch reconstruction silently drops real edits after a `Write`. In [`src/swebench_validate.py`](src/swebench_validate.py#L95), if a file has both `Write` and later `Edit` calls, the `full_write` branch wins and `changes['edits']` is ignored. Validation then runs against the wrong patch and can overwrite `runs.task_success` with a false result in [`src/swebench_validate.py`](src/swebench_validate.py#L265) and [`src/docker_validate.py`](src/docker_validate.py#L129).
- [ ] **P0-05** (cycle 2): Fake atomicity in storage. [`src/runner.py`](src/runner.py#L374) opens `BEGIN`, but [`src/db.py`](src/db.py#L197) and [`src/db.py`](src/db.py#L225) `commit()` inside `insert_run`/`insert_tool_call`. A mid-loop failure leaves partially committed `runs`/`tool_calls`, so counts and records can silently diverge.

## P1 Findings

- [ ] **P1-01** (cycle 1): Compound Bash commands are misclassified as read-only when the first token is read-only. In [`src/trace_collector.py`](src/trace_collector.py#L174), `|` gets special handling only if there is no `&&` or `;`. Then [`src/trace_collector.py`](src/trace_collector.py#L196) returns read-only based on the first token alone. Cases like `grep foo x && rm y` or `git diff; git apply p.diff` will be recorded as non-state-modifying and dropped from analysis.
- [ ] **P1-02** (cycle 1): `xargs` is treated as read-only in the base classifier. [`src/trace_collector.py`](src/trace_collector.py#L61) includes `xargs` in `READ_ONLY_COMMANDS`, and the single-command path at [`src/trace_collector.py`](src/trace_collector.py#L196) therefore classifies `xargs rm`, `xargs touch`, etc. as read-only. That undercounts state-modifying calls and weakens gating exactly on shell fan-out mutations.
- [ ] **P1-03** (cycle 1): Both validation scripts overwrite the primary `runs.task_success` label in place. [`src/docker_validate.py`](src/docker_validate.py#L127) and [`src/swebench_validate.py`](src/swebench_validate.py#L23) / [`src/swebench_validate.py`](src/swebench_validate.py#L140) mutate the same field the runner originally set. This silently destroys provenance between “workspace validation at run time” and “post hoc Docker/SWE-bench validation,” so downstream analysis cannot distinguish the original outcome from later re-evaluations.
- [ ] **P1-04** (cycle 1): Patch reconstruction for SWE-bench can silently produce the wrong diff. [`src/swebench_validate.py`](src/swebench_validate.py#L105) replays `Edit` operations with `str.replace(old, new, 1)` and simply skips an edit if `old_string` is no longer present. Repeated substrings, shifted context from earlier edits, or mismatched edit targets yield corrupted or incomplete patches with no error, which can flip validation results and contaminate `task_success`.
- [ ] **P1-05** (cycle 13): The “atomic” run insert path is not atomic. [`src/runner.py:374`](file://src/runner.py#L374) and [`src/analysis.py:408`](file://src/analysis.py#L408) wrap writes in `BEGIN`, but [`src/db.py:218`](file://src/db.py#L218), [`src/db.py:247`](file://src/db.py#L247), and [`src/db.py:288`](file://src/db.py#L288) call `commit()` inside helpers. A later failure can leave partially written runs/tool_calls despite the caller believing rollback will restore consistency.
- [ ] **P1-06** (cycle 13): Patch extraction silently drops failed `Edit` replays. [`src/swebench_validate.py:102`](file://src/swebench_validate.py#L102) replays edits with `if old in modified: replace(...); else: no-op`. If the captured `old_string` mismatches, the change disappears from the generated patch with no error, biasing validation toward false negatives / “no patch extracted”.
- [ ] **P1-07** (cycle 13): Bash read-only classification has false negatives for mutating commands. In [`src/trace_collector.py:174`](file://src/trace_collector.py#L174), any pipeline containing `&&` or `;` skips pipeline analysis, then [`src/trace_collector.py:196`](file://src/trace_collector.py#L196) classifies by first token only. `cat a | tee b && rm c` is read-only because `cat` is whitelisted. Also `xargs` is globally whitelisted at [`src/trace_collector.py:61`](file://src/trace_collector.py#L61), so plain `xargs rm < files` is misclassified read-only.
- [ ] **P1-08** (cycle 14): Patch reconstruction also silently drops failed `Edit` applications. [`src/swebench_validate.py`](src/swebench_validate.py#L106) does `replace(old, new, 1)` only if `old in modified`; otherwise it skips the edit with no error. That corrupts validation patches without any signal.
- [ ] **P1-09** (cycle 14): `prior_failure_streak` is biased downward by treating missing/unparseable tool results as success. [`src/analysis.py`](src/analysis.py#L103) and [`src/analysis.py`](src/analysis.py#L108) append `True` on missing/invalid `tool_result_json`. That silently erases preceding failures from the feature.
- [ ] **P1-10** (cycle 14): Standalone `xargs` is misclassified as read-only. [`src/trace_collector.py`](src/trace_collector.py#L61) includes `xargs` in `READ_ONLY_COMMANDS`, so `xargs rm`, `xargs mv`, etc. are counted as non-state-modifying unless they happen to be in a pipe caught by the separate pipe logic. This undercounts `total_state_modifying_calls` and drops decision records.
- [ ] **P1-11** (cycle 14): Unknown `git` subcommands default to read-only. [`src/trace_collector.py`](src/trace_collector.py#L217) returns read-only for any `git <subcommand>` not in `_GIT_MUTATING_SUBCOMMANDS`. Mutating commands omitted from that list are silently misclassified and excluded from analysis.
- [ ] **P1-12** (cycle 2): Reasoning is misattributed for mixed assistant messages. [`src/trace_collector.py`](src/trace_collector.py#L400) concatenates all text blocks in the message, then [`src/trace_collector.py`](src/trace_collector.py#L429) assigns that same `full_reasoning` to every tool call in the message. If content is `text -> tool_use -> text -> tool_use`, both calls get both texts. That corrupts per-call features and labels.
- [ ] **P1-13** (cycle 2): Bash read-only/state-modifying classification has concrete false negatives. [`src/trace_collector.py`](src/trace_collector.py#L52) marks `xargs` read-only, so standalone `xargs rm`/`xargs sed -i` is missed. [`src/trace_collector.py`](src/trace_collector.py#L108) treats `python -c`/`node -e` as read-only unless a short denylist hits; arbitrary mutating one-liners slip through. [`src/trace_collector.py`](src/trace_collector.py#L167) also misses redirections at command start like `> file`.
- [ ] **P1-14** (cycle 2): Validation patch extraction drops real changes. [`src/swebench_validate.py`](src/swebench_validate.py#L42) only replays `Write`/`Edit`, ignoring file mutations done via `Bash` (`sed -i`, `mv`, `patch`, heredocs, etc.), so successful runs can become `no patch extracted` / false unresolved in both SWE-bench and Docker validation.
- [ ] **P1-15** (cycle 2): Patch replay silently loses edits. [`src/swebench_validate.py`](src/swebench_validate.py#L106) applies each `Edit` only if `old_string in modified`; otherwise it skips with no error. Any context drift or repeated substring mismatch yields truncated patches and bad validation labels.
- [ ] **P1-16** (cycle 8): Atomicity is broken despite the explicit `BEGIN`: `insert_run()` and `insert_tool_call()` call `conn.commit()` internally, so [src/runner.py](src/runner.py#L374) and [src/analysis.py](src/analysis.py#L410) can leave partially written runs/tool_calls on mid-batch failure. The same bug is in [src/db.py](src/db.py#L197) and [src/db.py](src/db.py#L225).
- [ ] **P1-17** (cycle 8): Trace parsing silently drops malformed JSON lines and keeps going, which can undercount tool calls / lose records without marking the run corrupted. See [src/trace_collector.py](src/trace_collector.py#L334).
- [ ] **P1-18** (cycle 8): Bash classification has false negatives for compound commands: `grep ... && rm ...` or `cat ... ; mv ...` skip the pipeline branch, then get classified by the first token only, so the mutating tail is missed. See [src/trace_collector.py](src/trace_collector.py#L172) and [src/trace_collector.py](src/trace_collector.py#L196).
- [ ] **P1-19** (cycle 8): Bash classification also mislabels writes using redirection without trailing whitespace, e.g. `echo hi >out.txt` / `>>out.txt`; the regex misses them, then read-only commands like `echo` are accepted as non-mutating. See [src/trace_collector.py](src/trace_collector.py#L165) and [src/trace_collector.py](src/trace_collector.py#L196).
- [ ] **P1-20** (cycle 8): `xargs` is globally whitelisted as read-only, so `xargs rm`, `xargs sed -i`, etc. are recorded as non-state-modifying. See [src/trace_collector.py](src/trace_collector.py#L60) and [src/trace_collector.py](src/trace_collector.py#L196).
- [ ] **P1-21** (cycle 8): Validation reconstructs patches from only `Write`/`Edit` events, ignoring file mutations done through state-modifying `Bash` (`sed -i`, `mv`, `patch`, shell redirection, etc.). That can mark a real fix unresolved and overwrite `task_success` with a false failure. See [src/swebench_validate.py](src/swebench_validate.py#L23) and [src/docker_validate.py](src/docker_validate.py#L98).
- [ ] **P1-22** (cycle 8): `call_level_correlations()` computes Spearman stats and apparent sample sizes on per-call rows while the label is run-level `task_success`; repeated calls from the same run violate independence, so the reported rho/p-values are invalid/inflated even though the code prints them. See [src/analysis.py](src/analysis.py#L294).
- [ ] **P1-23** (cycle 9): The “single transaction” guarantees are broken. `runner.py` and `analysis.py` call `BEGIN`/`ROLLBACK`, but the DB helpers commit inside each insert/update, so a mid-run failure can leave a persisted `runs` row plus a partial subset of `tool_calls` or features already committed. Silent partial writes become unrecoverable state, not atomic failures. [src/db.py](src/db.py#L197) [src/db.py](src/db.py#L225) [src/db.py](src/db.py#L258) [src/runner.py](src/runner.py#L373) [src/analysis.py](src/analysis.py#L413)
- [ ] **P1-24** (cycle 9): `prior_failure_streak` silently undercounts failures. Missing or unparseable `tool_result_json` is treated as success, so dropped tool results and parse errors reset the streak instead of preserving uncertainty/failure. That corrupts a core feature without surfacing an error. [src/analysis.py](src/analysis.py#L99)
- [ ] **P1-25** (cycle 9): Validation can silently score the wrong patch. `extract_patch_from_stream()` only replays `Write`/`Edit`, ignores state-modifying `Bash`, and for `Edit` silently skips an edit if `old_string` is not found. That can produce incomplete patches, dropped file changes, and wrong `task_success` labels in both Docker and SWE-bench validation. [src/swebench_validate.py](src/swebench_validate.py#L23) [src/swebench_validate.py](src/swebench_validate.py#L102) [src/docker_validate.py](src/docker_validate.py#L98)
- [ ] **P1-26** (cycle 9): Bash state-modifying classification is wrong for compound commands. The classifier only handles pipes specially; `&&`/`;` chains fall through to first-token logic, so commands like `cat foo && rm foo` or `git diff && git apply patch` can be marked read-only and never recorded/gated. Standalone `xargs` is also whitelisted as read-only even though it commonly executes mutating commands. ## P2 Findings

- [ ] **P2-01** (cycle 1): Feature extraction commits one row at a time. [`src/db.py`](src/db.py#L258) commits inside `update_features()`, and [`src/analysis.py`](src/analysis.py#L157) calls it per tool call. An interrupted analysis run leaves mixed old/new feature values within the same dataset, which is a silent analysis-integrity problem for correlations and summaries.
- [ ] **P2-02** (cycle 13): Analysis summary overstates analyzed sample size. [`src/analysis.py:168`](file://src/analysis.py#L168) counts all SWE-bench runs, but pass/fail sums exclude `NULL task_success`; the DB currently has 3 such runs. The printed denominator and returned `n_runs` include unlabeled runs, so the summary is internally inconsistent and can misstate analysis coverage.
- [ ] **P2-03** (cycle 14): Task-level analysis excludes runs with zero extracted `tool_calls`. [`src/analysis.py`](src/analysis.py#L199) inner-joins `tool_calls` to `runs`, so runs with no state-modifying calls disappear entirely from correlation analysis. That biases task-level aggregates toward trajectories that already made state changes.
- [ ] **P2-04** (cycle 2): Call-level analysis is statistically invalid but still emitted as a result. [`src/analysis.py`](src/analysis.py#L299) repeats run-level `task_success` onto every call and then computes Spearman/p-values over call rows at [`src/analysis.py`](src/analysis.py#L352). The warning at [`src/analysis.py`](src/analysis.py#L329) is correct: n and significance are inflated, so those reported correlations are not valid inference.
- [ ] **P2-05** (cycle 2): `prior_failure_streak` is biased toward undercounting failures. [`src/analysis.py`](src/analysis.py#L103) treats unparseable or missing `tool_result_json` as success at [`src/analysis.py`](src/analysis.py#L109) and [`src/analysis.py`](src/analysis.py#L111). Any dropped/malformed tool result silently suppresses failure streaks.
- [ ] **P2-06** (cycle 8): `extract_features_for_all()` treats missing/unparseable `tool_result_json` as success when building `prior_failure_streak`, biasing failure-streak downward after any parse/result loss. See [src/analysis.py](src/analysis.py#L99).
- [ ] **P2-07** (cycle 8): Retry detection keys Bash calls by the first 80 chars of `command`, so distinct commands with the same prefix collapse together and inflate retry counts. See [src/analysis.py](src/analysis.py#L116).
- [ ] **P2-08** (cycle 9): Read-only `git fetch` is classified as state-modifying. `_GIT_MUTATING_SUBCOMMANDS` includes `fetch`, which mutates local refs but not the task workspace. If your taxonomy is “bash read-only vs state-modifying” in the sense of repository/filesystem risk, this overcounts state-modifying calls and contaminates the gating label. [src/trace_collector.py](src/trace_collector.py#L82)
- [ ] **P2-09** (cycle 9): The current DB looks internally consistent on coarse counts only. `SUM(total_state_modifying_calls)=COUNT(tool_calls)=1345`, and no run has `total_tool_calls < total_state_modifying_calls`; that does not catch the partial-write bug above because helper-level commits can still leave logically incomplete runs that satisfy those invariants.
